# Рекомендации по безопасности для UE-V 1.0


В этой статье содержится краткий обзор учетных записей и групп, файлов журналов и других вопросов, связанных с безопасностью, для Microsoft User Experience Virtualization (UE-V). Чтобы получить дополнительные сведения, воспользуйтесь приведенными ниже ссылками.

## Рекомендации по безопасности для конфигурации UE-V


**При создании общего доступа к хранилищу параметров Ограничьте доступ пользователей к общему доступу.**

-   Ограничьте общий доступ только для тех пользователей, которым нужен доступ. Создайте группу безопасности для пользователей с перенаправленными папками в определенном общем доступе и ограничьте доступ только для тех пользователей.

-   После создания общего доступа Скройте общий доступ, указав $ после имени общего доступа. Это приведет к скрытию общего доступа в обычных браузерах, и общий доступ не будет виден в окне Сетевое окружение.

### Использование серверов Windows Server 2003 или более поздних версий для размещения перенаправленных файловых ресурсов

Файлы пакета параметров пользователя содержат персональные данные, которые передаются между клиентским компьютером и сервером, на котором хранятся пакеты параметров. По этой причине вы должны убедиться, что данные защищены при пересылке по сети.

Данные параметров пользователя уязвимы для этих потенциальных угроз: перехват данных по мере их появления в сети; изменения данных, передаваемых по сети; и подмена сервера, на котором размещены данные.

Некоторые функции Windows Server 2003 и более поздних версий помогут защитить пользовательские данные.

-   **Kerberos** — Стандартный на всех версиях Windows 2000 и windows Server 2003 и более поздних версий. Kerberos обеспечивает наивысший уровень безопасности сетевых ресурсов. NTLM проверяет подлинность клиента; Kerberos проверяет подлинность сервера и клиента. При использовании NTLM клиент не знает, является ли сервер допустимым. Это особенно важно, если клиент обменивается личными файлами с сервером, как в случае с перемещаемыми профилями. Протокол Kerberos обеспечивает более высокую безопасность, чем NTLM. Протокол Kerberos недоступен в операционных системах Windows NT версии 4,0 и более ранних версий.

-   **IPSec** — протокол IP-безопасности (IPSec) обеспечивает проверку подлинности на уровне сети, целостность данных и шифрование. IPsec обеспечивает следующее:

    -   Перемещаемые данные изменяются при изменении данных во время прохождения по пути.

    -   Перемещенные данные можно безопасно отсекать, просматривать и копировать.

    -   Перемещаемые данные безопасно получать от пользователей, не прошедших проверку подлинности.

-   **Подписывание SMB** — протокол проверки подлинности сообщений сервера (SMB) поддерживает проверку подлинности, что предотвращает запуск активного сообщения и атаки "злоумышленник в середине". Подписывание SMB обеспечивает такую проверку подлинности, помещая цифровую подпись в каждый из протоколов SMB. После этого цифровая подпись будет проверяться клиентом и сервером. Чтобы использовать подписывание SMB, необходимо сначала включить его или потребовать его для клиента SMB и сервера SMB. Обратите внимание, что при подписывании SMB накладывается снижение производительности. Она не потребляет больше пропускной способности сети, но использует больше тактов ЦП на стороне клиента и сервера.

### Всегда используйте файловую систему NTFS для томов, содержащих данные пользователей.

Для наиболее безопасной конфигурации настройте серверы, на которых размещены файлы параметров UE-V, для использования файловой системы NTFS. В отличие от файловой системы FAT, NTFS поддерживает избирательные списки управления доступом (DACL) и списки управления доступом системы (SACL). Списки DACL и SACL управляют тем, кто может выполнять операции с файлом, а также события, которые будут запускать ведение журнала действий, выполняемых с файлом.

### <a href="" id="do-not-rely-on-efs-to-encrypt-users--files-when-transmitted-over-the-network"></a>Не полагаться на шифрование EFS для шифрования файлов пользователей при передаче по сети

При использовании шифрованной файловой системы (EFS) для шифрования файлов на удаленном сервере зашифрованные данные не шифруются при передаче по сети; Оно шифруется только при сохранении на диске.

Исключением является тот момент, когда система включает IP-безопасность (IPsec) или веб-сайт для создания и управления версиями (WebDAV). IPsec шифрует данные при транспортировке в сети TCP/IP. Если файл зашифрован и не копируется в папку WebDAV на сервере, он останется зашифрованным во время передачи и хранится на сервере.

### Шифрование кэша автономных файлов

По умолчанию кэш автономных файлов защищен для разделов NTFS с помощью ACL, но шифрование кэша дополнительно повышает безопасность на локальном компьютере. По умолчанию кэш на локальном компьютере не зашифрован, поэтому любые зашифрованные файлы, кэшированные из сети, не будут зашифрованы на локальном компьютере. Это может представлять угрозу безопасности в некоторых средах.

Если включено шифрование, все файлы в кэше автономных файлов шифруются. Сюда входит шифрование существующих файлов и файлов, которые были добавлены позже. На локальном компьютере влияет кэшированная копия, но связанная с ней сетевая копия отсутствует.

Кэш можно зашифровать одним из двух способов:

1.  С помощью групповой политики. -Включите параметр **шифрование кэша автономных файлов** , расположенный на компьютере Configuration\\Administrative Templates\\Network\\Offline Files в редакторе групповой политики.

2.  Устанавливать. -Выберите Инструменты, а затем — Параметры папки в меню команд проводника Windows. Выберите вкладку Автономные файлы, а затем установите флажок **Шифровать автономные файлы для защиты данных** .

### Разрешите агенту UE-V создавать папки для каждого пользователя

Чтобы обеспечить оптимальное функционирование UE-V, создайте только корневой общий доступ на сервере и предоставьте агенту UE-V создание папок для каждого пользователя. UE-V создаст эти папки пользователей с соответствующей защитой.

С помощью этой конфигурации разрешений пользователи могут создавать папки для хранения параметров. Агент UE-V создает и защищает папку settingspackage во время работы в контексте пользователя. Пользователь получает полный доступ к своей папке settingspackage. Другие пользователи не наследуют доступ к этой папке. Вам не нужно создавать и защищать отдельные каталоги пользователей. Это будет выполнено автоматически агентом, который запускается в контексте пользователя.

**Примечание.**  
Дополнительную защиту можно настроить, когда используется сервер Windows для общего доступа к хранилищу параметров. UE-V можно настроить так, чтобы убедиться в том, что группа локального администратора или текущий пользователь является владельцем папки, в которой хранятся пакеты параметров. Чтобы включить дополнительную защиту, используйте следующую команду:

1.  Добавьте раздел реестра REG \ _DWORD с именем "RepositoryOwnerCheckEnabled" `HKEY_LOCAL_MACHINE\Software\Microsoft\UEV\Agent\Configuration` .

2.  Задайте для раздела реестра значение 1.

Если этот параметр конфигурации установлен, агент UE-V Agent удостоверяет, что локальная группа администраторов или текущий пользователь является владельцем папки settingspackage. В противном случае агент UE-V не будет разрешать доступ к папке.



Если необходимо создать папки для пользователей и убедиться, что у вас установлены правильные разрешения.

Настоятельно рекомендуется не создавать папки, а это значит, что вы разрешаете агенту UE-V создать папку для пользователя.

### <a href="" id="ensure-that-correct-permissions-are-set-when-storing-ue-v-settings-in-a-user-s-home-directory"></a>Убедитесь в том, что при сохранении параметров UE-V в основном каталоге пользователя будут заданы правильные разрешения

Если вы перенаправляли параметры UE-V в основной каталог пользователя, проверьте, правильно ли настроены разрешения для основного каталога пользователя в Организации.
